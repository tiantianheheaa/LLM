这是因为 Qwen3 的 `generation_config.json` 中**同时把 151643 和 151645 都配置为合法的 `eos_token_id`**，而 beam search 的每条路径独立采样，遇到任一 EOS 都会停止，因此不同 beam 可能命中不同的结束标记。

具体对应关系如下：


| **151643** | |<|endoftext|> | 通用文本结束符 / 部分模型的 PAD token |

| **151645** | |<|im_end|>（即 `im_end`） | **对话消息结束符**，Chat/Instruct 模型的"正确"停止标记 |

**根本原因：**

1. **双 EOS 配置**：Qwen3 系列（包括 0.6B）的 `generation_config.json` 中 `eos_token_id` 通常配置为数组 `[151645, 151643]`。这意味着生成器认为遇到其中**任意一个** token 都算正常结束。

2. **Beam Search 的并行路径特性**：`beam=16` 时，16 条候选路径各自独立解码、各自维护 logits 分布。某些 beam 在特定上下文中对 `151645` 的概率最高，于是以 `|<|im_end|>` 结束；另一些 beam 可能在不同位置对 `151643` 的概率突增，于是以 `|<|endoftext|>` 结束。

3. **模型规模的影响**：0.6B 小模型对结束符的预测稳定性较弱，更容易在 beam 之间产生分歧。大模型（如 8B、30B）的 logits 分布更集中，绝大多数 beam 都会统一在 `151645` 结束。

**实际影响：**

- 以 **151645** (`<|<|im_end|>`) 结尾：符合 Qwen3 Chat 模板预期，表示 assistant 消息完整结束。
- 以 **151643** (`<|<|endoftext|>`) 结尾：虽然也是合法停止，但通常意味着生成提前中断，可能导致输出不完整（类似 vLLM 社区报告的截断问题）。

**建议做法：**

如果你希望所有 beam 统一在对话结束符处停止，可以在调用时显式指定 `eos_token_id=151645`（单值），或把 `151643` 加入 `bad_words_ids` 进行屏蔽，让模型只认 `|<|im_end|>` 作为唯一停止信号。
