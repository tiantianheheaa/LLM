# 简单版
我来解释这几组核心计算资源的概念及其联系。

## 一、CPU 相关概念

| 概念 | 含义 | 作用 |
|------|------|------|
| **CPU** | 中央处理器，通用计算核心 | 负责逻辑控制、任务调度、串行计算，是系统的"大脑" |
| **内存 (RAM)** | 临时数据存储，断电丢失 | 存放正在运行的程序和数据，供CPU高速读写 |
| **内存带宽** | 内存与CPU之间的数据传输速率 | 决定CPU能以多快的速度获取/写入数据 |

**关系**：CPU算得快，但如果内存带宽不足，CPU会"饿着肚子等数据"，形成**内存墙**瓶颈。

---

## 二、GPU 相关概念

| 概念 | 含义 | 作用 |
|------|------|------|
| **GPU** | 图形处理器，大规模并行计算核心 | 擅长同时执行成千上万个简单计算（如矩阵运算） |
| **显存 (VRAM)** | GPU专用的内存 | 存放模型参数、图像数据、计算中间结果 |
| **显存带宽** | 显存与GPU核心之间的数据传输速率 | 决定GPU能以多快的速度读取/写入显存数据 |

**关系**：GPU核心数多、算力极强，但显存带宽往往成为瓶颈——尤其是AI大模型推理时，参数需要从显存频繁搬运到计算单元。

---

## 三、核心联系与对比

```
┌─────────────────────────────────────────┐
│  CPU  ←────内存带宽────→  内存(RAM)    │
│   ↑                                    │
│   │  通过PCIe总线（带宽更低）           │
│   ↓                                    │
│  GPU  ←──显存带宽────→  显存(VRAM)    │
└─────────────────────────────────────────┘
```

### 关键联系点：

1. **带宽决定利用率**
   - 算力再强，带宽不够，计算单元就会空闲等待。例如：GPT-4推理时，90%时间花在等数据从显存搬到计算单元。
   - ** roofline模型**：性能上限 = min(算力峰值, 带宽 × 计算强度)

2. **显存 vs 内存的定位差异**
   - **显存**：紧耦合，带宽极高（H100达3.35TB/s），但容量小（40-80GB）
   - **内存**：容量大（TB级），但带宽低（DDR5约50-100GB/s），且离GPU远

3. **PCIe 是瓶颈桥接**
   - CPU内存 ↔ GPU显存 通过PCIe传输，带宽仅几十GB/s，远低于显存带宽。
   - 因此**数据搬运成本极高**，算法设计要尽量让数据"待在显存里算完"。

4. **应用场景决定瓶颈**
   - **大模型推理**：瓶颈通常是**显存容量**（装不下模型）和**显存带宽**（参数读取速度）
   - **训练**：瓶颈通常是**算力**和**显存容量**
   - **数据处理**：瓶颈通常是**内存带宽**或**磁盘I/O**

---

## 四、形象比喻

| 组件 | 比喻 |
|------|------|
| CPU/GPU | 工厂里的工人（GPU是1000个简单工人，CPU是8个高级工程师） |
| 内存/显存 | 仓库，存放原材料和半成品 |
| 带宽 | 仓库到工位的传送带速度 |

**核心洞察**：现代AI系统的性能优化，本质上是在解决**"传送带不够快"**和**"仓库不够大"**的问题——这也是为什么HBM（高带宽显存）和模型量化（压缩参数）技术如此关键。

# 详细版

我来详细解释这些核心计算资源的概念，并以LLM训练为实例深入分析它们之间的联系。

---

## 一、基础概念详解

### 1. CPU（中央处理器）

**定义**：通用计算核心，通常包含少量高性能核心（如8-128核），擅长复杂逻辑控制、分支预测、串行任务和系统调度。

**关键特征**：
- **低延迟**：单线程响应快，适合需要复杂决策的任务
- **强控制流**：擅长`if/else`、循环、指针跳转等不规则计算
- **缓存层级复杂**：L1/L2/L3缓存体系，优化数据局部性

**在AI系统中的角色**：
- 数据预处理与加载（读取磁盘、解码、数据增强）
- 训练流程控制（迭代管理、checkpoint保存、日志记录）
- 分布式协调（AllReduce通信调度、任务分配）
- 无法直接参与大规模矩阵运算（效率太低）

---

### 2. 内存（RAM，主存）

**定义**：CPU可直接寻址的临时存储，存放操作系统、应用程序和活跃数据，断电后数据丢失。

**关键参数**：
| 参数 | 典型值 | 意义 |
|------|--------|------|
| 容量 | 512GB - 2TB（服务器级） | 决定能同时处理多少数据 |
| 类型 | DDR4/DDR5 | DDR5带宽更高、功耗更低 |
| 带宽 | 50-100 GB/s（单路） | 内存与CPU之间的传输速度 |
| 延迟 | 80-100 ns | 从CPU请求到数据到达的时间 |

**内存层次结构**：
```
寄存器（<1ns）→ L1缓存（1-2ns）→ L2缓存（3-10ns）→ L3缓存（10-20ns）→ 内存（80-100ns）→ 磁盘（ms级）
```
越靠近CPU，速度越快、容量越小、成本越高。

---

### 3. 内存带宽

**定义**：单位时间内内存与CPU之间能传输的数据量，通常以 GB/s 为单位。

**计算公式**：
```
内存带宽 = 内存频率 × 总线宽度 × 通道数 × 每周期传输次数
```
例如：DDR5-4800，8通道，64bit位宽：
```
4800 MT/s × 8 Byte × 8 通道 = 307.2 GB/s（理论峰值）
```

**实际瓶颈**：由于缓存未命中、预取失败、地址冲突等，**实际有效带宽通常只有理论值的60-70%**。

---

### 4. GPU（图形处理器/加速器）

**定义**：大规模并行计算架构，包含数千个简单计算核心，专为吞吐型计算设计。

**关键特征**：
| 特征 | CPU | GPU | 说明 |
|------|-----|-----|------|
| 核心数 | 8-128 | 10,000+ | GPU核心极简单，只能做乘加 |
| 线程管理 | 复杂分支预测 | SIMT（单指令多线程） | GPU要求线程执行相同指令 |
| 设计理念 | 低延迟 | 高吞吐 | GPU用延迟换吞吐，隐藏内存延迟 |
| 适合计算 | 不规则、串行 | 规则、数据并行 | 矩阵乘法完美匹配GPU |

**现代AI GPU（以NVIDIA H100为例）**：
- 16,896个CUDA核心 + 528个Tensor Core
- FP16算力：989 TFLOPS（Tensor Core）
- 专为矩阵乘法和卷积优化

---

### 5. 显存（VRAM，GPU Memory）

**定义**：GPU专用的物理内存，与GPU核心封装在同一芯片或紧邻的PCB上。

**技术演进**：
| 技术 | 带宽 | 容量 | 应用 |
|------|------|------|------|
| GDDR6/X | 1 TB/s | 24-48 GB | 消费级显卡（RTX 4090） |
| HBM2e | 1.6 TB/s | 40-80 GB | 数据中心（A100） |
| HBM3 | 3.35 TB/s | 80 GB | 最新数据中心（H100） |
| HBM3e | 5 TB/s+ | 192 GB | 下一代（B100/H200） |

**显存 vs 内存核心差异**：
- **物理距离**：显存紧贴GPU核心（2.5D/3D封装），内存通过主板走线连接CPU
- **位宽**：HBM采用4096-bit超宽总线，DDR5仅64-bit
- **延迟**：显存延迟反而比内存高（~300ns vs ~100ns），但**带宽高30-50倍**

---

### 6. 显存带宽

**定义**：GPU核心与显存之间的数据传输速率，是GPU架构中最关键的性能指标之一。

**为什么显存带宽如此重要**：
GPU的算力（TFLOPS）增长远快于显存带宽增长。以H100为例：
- 算力：989 TFLOPS
- 显存带宽：3.35 TB/s = 3350 GB/s

**算术强度（Operational Intensity）**：
```
所需带宽 = 算力 / 算术强度
```
对于矩阵乘法 $C = A \times B$，若矩阵大小为 $N \times N$，计算量为 $2N^3$，数据量为 $3N^2$（读A、B，写C），算术强度为 $2N^3 / 3N^2 = 2N/3$。

当 $N$ 很大时，算术强度高，计算-bound；当 $N$ 小或层间通信多时，带宽-bound。

---

## 二、系统架构与数据流全景

```
┌─────────────────────────────────────────────────────────────┐
│                      存储层级（容量大→小，速度慢→快）          │
│  磁盘/SSD  ←──网络存储──→  内存(RAM)  ←──PCIe──→  显存(VRAM) │
│  (TB级, MB/s)             (TB级, GB/s)    (32GB/s)   (GB级, TB/s)│
│       ↑                                              ↓       │
│       └──────── CPU 数据预处理 ─────────────────── GPU 计算 ──┘
│              (Python/PyTorch DataLoader)       (CUDA Kernel)
└─────────────────────────────────────────────────────────────┘
```

**关键互联瓶颈**：
| 互联 | 带宽 | 延迟 | 瓶颈场景 |
|------|------|------|----------|
| CPU↔内存 | 100 GB/s | 100 ns | 大数据预处理 |
| CPU↔GPU (PCIe 4.0 x16) | 32 GB/s | 1-2 μs | 数据上载、梯度回传 |
| GPU↔显存 | 3350 GB/s | 300 ns | 模型参数读取 |
| GPU↔GPU (NVLink) | 900 GB/s | 亚μs | 多卡通信、分布式训练 |

## 三、以LLM训练为例的深度分析

### 场景设定：训练 GPT-3 规模模型（175B参数）

**模型配置**：
- 参数：1750亿（175B）
- 精度：FP16（2字节/参数）
- 模型状态：参数 + 梯度 + 优化器状态（Adam：动量 + 二阶矩）
- **显存占用**：$175B \times 2 \times 4 = 1.4 TB$（理论值，实际因并行策略不同）

**训练配置**：
- 序列长度：2048 tokens
- 批量大小：每卡 micro-batch=1，全局batch通过梯度累积或数据并行扩展
- 硬件：8× H100 SXM（80GB显存/卡）

---

### 1. 训练流程中的组件分工

#### **CPU 的职责**

| 阶段 | 具体工作 | 性能影响 |
|------|----------|----------|
| **数据加载** | 从磁盘/NFS读取原始文本，进行Tokenizer编码（BPE/WordPiece） | 若CPU核心不足，GPU会空闲等待数据（**CPU瓶颈**） |
| **数据预处理** | 填充(padding)、掩码(mask)生成、位置编码 | 需内存带宽支持快速张量操作 |
| **DataLoader** | 多进程（`num_workers`）将数据从页锁定内存(pin_memory)异步传输到GPU | 需要CPU↔内存高带宽，以及PCIe带宽 |
| **训练控制** | 管理训练循环、验证、checkpoint保存、学习率调度 | 轻量级，但checkpoint写入时会阻塞 |
| **分布式协调** | 通过NCCL库协调多卡通信（AllReduce、AllGather） | 通信延迟影响扩展效率 |

**CPU瓶颈实例**：
若使用消费级CPU（如AMD Ryzen）配8张H100：
- CPU核心数少（16核），无法并行处理8路数据流
- 内存带宽仅50 GB/s，8个DataLoader竞争带宽
- 结果：GPU利用率仅30-50%，大量时间等待数据

**优化方案**：
- 使用高核心数服务器CPU（AMD EPYC 96核+）
- 配置大容量内存（2TB+），启用`pin_memory=True`
- 使用NVMe SSD + `mmap`模式减少IO等待
- 采用`torch.compile`或`DALI`库将预处理卸载到GPU

---

#### **内存（RAM）的职责**

| 用途 | 占用估算 | 说明 |
|------|----------|------|
| 操作系统 + PyTorch进程 | 50-100 GB | 多进程训练时随卡数倍增 |
| 数据缓冲区 | 100-500 GB | `prefetch_factor` × batch_size，避免GPU饥饿 |
| 页锁定内存 (pin_memory) | 50-200 GB | CPU→GPU异步传输的跳板，必须驻留物理内存 |
| Checkpoint缓存 | 200-400 GB | FP32完整模型状态，用于故障恢复 |
| 激活值检查点（可选） | 100-300 GB | 若启用CPU offloading，将GPU激活值换入内存 |

**内存带宽瓶颈**：
当启用**ZeRO-Offload**（将优化器状态卸载到CPU内存）：
- 每步需将梯度从GPU→CPU（PCIe 32 GB/s）
- CPU计算更新后，将参数从CPU→GPU
- **PCIe带宽成为瓶颈**：175B参数的FP32梯度 = 350 GB，传输需 ~11秒，完全不可接受

**优化**：仅卸载优化器状态（ZeRO-2），不卸载梯度，减少通信量。

---

#### **内存带宽的实际影响**

在纯CPU预处理场景中：
- Tokenizer编码是内存密集型操作（大量字符串→整数映射）
- 内存带宽100 GB/s意味着每秒可处理约50GB原始文本（假设2倍开销）
- 对于1T token数据集，预处理需数小时，成为训练启动瓶颈

---

#### **GPU 的计算角色**

**前向传播（Forward）**：
```
输入: [batch_size, seq_len] token IDs
     ↓ Embedding层（查表操作，显存带宽 intensive）
     ↓ Transformer Block × 96层
       ├─ LayerNorm（element-wise，计算轻，带宽 intensive）
       ├─ QKV Projection: [B,S,12288] × [12288, 3×12288] → 矩阵乘，Tensor Core
       ├─ Attention: Q×K^T, Softmax, ×V → 内存带宽 intensive（需频繁读写显存）
       ├─ MLP: [B,S,12288] × [12288, 4×12288] × [4×12288, 12288] → 矩阵乘
       └─ 残差连接（element-wise，带宽 intensive）
     ↓ LayerNorm + LM Head（[B,S,12288] × [12288, vocab_size]）
输出: logits [B,S,50257]
```

**计算特征分析**：
| 操作 | 算力需求 | 带宽需求 | 瓶颈类型 |
|------|----------|----------|----------|
| 大矩阵乘法（MLP, QKV） | 极高 | 中 | **计算瓶颈**（Tensor Core饱和） |
| Attention中的Softmax | 低 | 高 | **带宽瓶颈**（需读写中间结果） |
| LayerNorm, Dropout | 低 | 极高 | **带宽瓶颈**（element-wise，无计算隐藏延迟） |
| Embedding查表 | 无 | 极高 | **纯带宽瓶颈** |

**关键洞察**：LLM训练中，**约30-40%时间花在带宽密集型操作**（Attention、归一化、激活函数），而非纯矩阵乘法。

---

#### **显存（VRAM）的关键约束**

**175B模型在单卡H100（80GB）上的显存分布**：

| 组件 | 显存占用 | 说明 |
|------|----------|------|
| 模型参数（FP16） | 350 GB | 无法放入单卡！必须模型并行 |
| 梯度（FP16） | 350 GB |  |
| 优化器状态（FP32） | 700 GB | Adam的动量和二阶矩 |
| 激活值 | 50-200 GB | 取决于序列长度和是否重计算 |
| **总计** | **1.4 TB+** | **单卡80GB vs 需求1.4TB = 18×差距** |

**并行策略对显存的影响**：

1. **数据并行（DP）**：每卡存完整模型，显存无节省，仅加速数据加载
2. **张量并行（TP）**：将层内矩阵切分到多卡，每卡存部分参数。8卡TP可将参数显存降至1/8，但需在层内频繁通信（NVLink AllReduce）
3. **流水线并行（PP）**：将模型按层切分到多卡，每卡存几层。减少单卡参数，但引入流水线气泡（bubble）
4. **ZeRO（优化器状态分片）**：将优化器状态分散到所有数据并行卡上，单卡显存节省与数据并行度成正比

**实际配置（8×H100训练175B模型）**：
- TP=4（单节点内NVLink高速互联）+ PP=4（节点间）+ DP=2
- 或采用 ZeRO-3 + 梯度检查点 + FP16/BF16混合精度
- 单卡显存占用压缩至60-70GB，刚好适配80GB显存

---

#### **显存带宽的决定性作用**

**训练中的显存带宽消耗**：

每步训练（1个micro-batch）中，显存带宽用于：
1. **读取参数**：从显存加载到寄存器/共享内存进行计算
2. **写入梯度**：反向传播时写回显存
3. **激活值读写**：前向保存/反向读取激活值（若未重计算）
4. **优化器状态读写**：参数更新时读取/写入动量和二阶矩

**带宽需求量化**：
假设每步处理2048 tokens，175B模型：
- 前向计算量：~350 TFLOPS（每token）
- 反向计算量：~700 TFLOPS（约2×前向）
- 总计算：~2 PFLOPS（每步每token）

H100算力989 TFLOPS，理论需2秒完成计算。
但显存带宽3.35 TB/s，每步需搬运：
- 参数读取：350 GB × 2（前向+反向）= 700 GB
- 梯度读写：350 GB
- 激活值：100 GB（假设重计算）
- 优化器状态：700 GB（若未offload）

总计约1.5 TB数据搬运，需时：$1.5TB / 3.35TB/s \approx 0.45s$

**算术强度分析**：
```
算术强度 = 总计算量 / 总数据搬运 = 2 PFLOPS / 1.5 TB = 1.3 FLOP/Byte
```
对于H100，要达到算力峰值需要算术强度 > 100 FLOP/Byte（Tensor Core）。
实际1.3远低于此，说明**训练严重受限于显存带宽，而非算力**。

** roofline模型可视化**：
```
性能 (TFLOPS)
    │
989 ├─────────────────────────────── 算力峰值（水平线）
    │                    ╱
    │                 ╱
335 ├──────────────╱──────────────── 带宽限制线（斜率=带宽）
    │           ╱  ↑
    │        ╱     │ 实际运行点（算术强度低，落在带宽线）
    │     ╱        │
    │  ╱           │
    └───────────────────────────────
         1   10   100  算术强度 (FLOP/Byte)
```

### 2. 多维度瓶颈分析

#### **场景A：单卡训练小模型（7B参数）**

| 资源 | 状态 | 瓶颈？ |
|------|------|--------|
| GPU算力 | 使用30% | 否，算力过剩 |
| 显存容量 | 占用14GB/80GB | 否，充足 |
| 显存带宽 | 饱和90% | **是，主要瓶颈** |
| CPU | 轻度使用 | 否 |
| 内存带宽 | 使用20% | 否 |

**现象**：GPU利用率低（30%），但显存带宽接近饱和。这是因为小模型算术强度低，每Byte数据只贡献少量计算，GPU核心大量空闲等待数据。

**优化**：增大batch size（提高算术强度），或使用梯度累积虚拟扩大batch。

---

#### **场景B：多卡数据并行训练（32卡，数据并行）**

| 资源 | 状态 | 瓶颈？ |
|------|------|--------|
| GPU算力 | 使用85% | 轻微瓶颈 |
| 显存带宽 | 饱和 | **是** |
| NVLink | 通信占用20%带宽 | 轻微瓶颈 |
| CPU | 32个DataLoader进程竞争 | **是，可能瓶颈** |
| 内存带宽 | 饱和 | **是，主要瓶颈** |

**现象**：32个进程同时从内存读取数据，内存带宽（100 GB/s）被瓜分，每进程仅3 GB/s，导致GPU饥饿。

**优化**：使用分布式文件系统（如Lustre）+ 本地NVMe缓存，减少内存压力；或减少`num_workers`，增加`prefetch_factor`。

---

#### **场景C：超大模型流水线并行（175B，64卡）**

| 资源 | 状态 | 瓶颈？ |
|------|------|--------|
| GPU算力 | 使用60% | 否，流水线气泡 |
| 显存带宽 | 饱和 | 是 |
| NVLink/IB | 频繁PP通信 | **是，主要瓶颈** |
| CPU | 轻量 | 否 |
| 内存 | 仅用于checkpoint | 否 |

**现象**：流水线并行的**气泡（bubble）**导致GPU空闲。假设PP=8，每步需填充和排空流水线，理想利用率 = $(N-1)/N = 87.5\%$，实际因负载不均更低。

**优化**：采用**交错流水线（interleaved pipeline）**，将每层拆分到多阶段，减少气泡；或使用**GPipe**的micro-batch填充策略。

### 3. 关键优化技术及其对资源的影响

| 技术 | 目标资源 | 原理 | 代价 |
|------|----------|------|------|
| **梯度检查点（Activation Checkpointing）** | 显存容量 | 前向不保存中间激活，反向时重计算 | 增加20-30%计算量，消耗更多显存带宽 |
| **ZeRO（分片优化器）** | 显存容量 | 将优化器状态分散到多卡CPU内存 | 增加通信量（PCIe/网络带宽瓶颈） |
| **混合精度（FP16/BF16）** | 显存容量+带宽 | 参数和激活用半精度，保留FP32主参数 | 数值稳定性风险，需loss scaling |
| **FlashAttention** | 显存带宽 | IO-aware算法，减少Attention的HBM读写 | 计算量增加，但总体更快（带宽瓶颈→计算瓶颈转移） |
| **张量并行（TP）** | 单卡显存 | 层内矩阵切分 | 增加NVLink通信，需高速互联 |
| **序列并行（SP）** | 显存容量 | 将序列维度切分到多卡 | 减少单卡激活值，增加通信 |
| **CPU Offload** | 显存容量 | 将不活跃状态换入内存 | **严重依赖PCIe带宽和内存带宽** |
| **量化训练（INT8/FP8）** | 显存容量+带宽 | 降低精度减少数据量 | 训练不稳定，需特殊硬件支持（H100 FP8 Tensor Core） |

**FlashAttention的深层意义**：
传统Attention的HBM访问：$O(N^2)$ 次读写（$N$=序列长度）。
FlashAttention通过分块（tiling）和重计算，将HBM访问降至 $O(N)$，虽然SRAM（共享内存）和寄存器计算增加，但**显著缓解显存带宽瓶颈**，使Attention从带宽-bound转为计算-bound。

### 4. 端到端训练性能模型

**训练吞吐量公式**（简化版）：

```
Tokens/s = min(
    GPU算力峰值 / (每token计算量),
    显存带宽 / (每token数据搬运量),
    PCIe带宽 / (CPU→GPU数据量),
    网络带宽 / (多卡通信量)
)
```

对于175B模型在8×H100上：
- **算力上限**：$8 \times 989 \text{ TFLOPS} / (350 \times 2 \text{ FLOP/token}) \approx 11,000 \text{ tokens/s}$
- **显存带宽上限**：$8 \times 3.35 \text{ TB/s} / (1.5 \text{ TB}/\text{step} \times \text{seq_len}) \approx 8,900 \text{ tokens/s}$
- **PCIe上限**：$32 \text{ GB/s} / (2 \text{ GB}/\text{batch}) \approx 16 \text{ batches/s} \rightarrow 32,000 \text{ tokens/s}$（若seq_len=2048）

**实际瓶颈**：显存带宽限制在 ~8,900 tokens/s，加上流水线气泡和通信开销，实际可能仅达 **5,000-6,000 tokens/s**。

**训练时间估算**：
- 1T tokens数据集，5,000 tokens/s：
- 总时间 = $10^{12} / 5000 / 3600 / 24 \approx 2,300$ 小时 ≈ **96天**
- 这还未计入验证、checkpoint、故障恢复时间。

## 四、总结：资源间的深层联系

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM训练的资源依赖图                        │
│                                                             │
│   数据流：磁盘 → 内存 → CPU预处理 → PCIe → 显存 → GPU计算    │
│              ↑         ↑              ↑         ↑           │
│              └─────────┴──────────────┴─────────┘           │
│                 瓶颈可能出现在任何环节                       │
│                                                             │
│   控制流：CPU调度 → GPU执行 → CPU收集指标 → 分布式协调        │
│                                                             │
│   状态流：GPU显存（参数/梯度/激活）↔ CPU内存（优化器/checkpoint）│
│                    ↑（PCIe瓶颈）                            │
└─────────────────────────────────────────────────────────────┘
```

### 核心联系总结

| 联系对 | 关系本质 | LLM训练中的体现 |
|--------|----------|----------------|
| **CPU ↔ 内存** | 预处理流水线 | CPU核心数和内存带宽决定数据供给速度，GPU是否饥饿 |
| **内存 ↔ GPU** | PCIe数据上载 | 每步需将batch数据从页锁定内存→显存，PCIe带宽限制batch size |
| **GPU ↔ 显存** | 核心计算循环 | 显存带宽是LLM训练的主要瓶颈（算术强度低），而非算力 |
| **显存容量 ↔ 模型规模** | 硬约束 | 模型>显存时必须并行或offload，引入通信/PCIe瓶颈 |
| **多GPU ↔ 网络** | 分布式扩展 | NVLink/InfiniBand带宽决定扩展效率，通信占比随卡数增加 |

### 设计哲学

现代LLM训练系统的优化是**多维资源平衡的艺术**：

1. **显存容量不够** → 模型并行/Offload → 引入通信/PCIe瓶颈
2. **显存带宽不够** → FlashAttention/量化 → 转移瓶颈到计算或稳定性
3. **PCIe带宽不够** → 减少CPU-GPU数据交换 → 增加显存压力
4. **CPU/内存不够** → GPU预处理（DALI）→ 增加GPU负担
5. **网络带宽不够** → 减少并行度 → 延长训练时间

**最终目标**：在成本约束下，找到使**最慢环节（瓶颈）尽可能不阻塞其他资源**的配置，即实现**资源均衡（balanced system）**——这也是构建AI集群时需要进行详细的roofline分析和性能建模的根本原因。
